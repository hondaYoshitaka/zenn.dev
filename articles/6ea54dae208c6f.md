ADR（Architecture Decision Record）、良い習慣だとわかっていても続かないですよね。
書くのが面倒だし、人によって粒度がバラバラだし、書いても埋もれる。

以前 [ADR について記事を書いた](https://qiita.com/hondaYoshitaka/items/a4ecedb31388ea2f8455#architectural-decision-recordsadr) のですが、大切さを説いたところで定着しなければ意味がありません。

この記事では、ADRを **書き続けられるしくみ** と **組織に届けるしくみ** の2つに絞って、私たちのチームで実際にやっていることを紹介します。

## 「書くのが面倒」を解決する — Claude Code skill による ADR 生成

ADRが定着しない最大の原因は「書くのが面倒」に尽きると思っています。技術選定の議論は盛り上がるのに、その結論をドキュメントに落とす段になるととたんにテンションが下がる。議事録すら残らないまま、決定だけがSlackの流れに消えていく。

であれば、書く部分はAIに任せてしまえばよい。人間は意思決定そのものに集中して、記録はClaude Codeのskillに任せる。そう考えてADR Creatorというskillを作りました。リポジトリの `.claude/skills/` に配置しているので、そのリポジトリで開発するメンバーなら誰でも使えます。

実行はこんな感じです。

```
/adr-creator xxxについてADRを書きたい
```

```
/adr-creator ここまでの議論をまとめてADRとして残したい
```

Claude Code skillの全量はGistに公開しています。

https://gist.github.com/hondaYoshitaka/83a40c59d670930a515fdaaf83755fdc

### ADR Creator のワークフロー

skillは以下の4ステップでADRを生成します。

| Step | やること | ポイント |
|------|---------|---------|
| 1. 情報収集 | 決定内容・背景・代替案・トレードオフ・参考リンクを確認 | すでに情報がそろっていれば不足分だけ聞く |
| 2. 連番の決定 | `docs/adr/` の既存ファイルから次の番号を自動決定 | 命名規則も skill が管理 |
| 3. ADR の執筆 | フォーマットとガイドラインに従って生成 | 比較表・コード例・緩和策を含む |
| 4. レビュー提示 | ドラフトを提示し、フィードバックを反映 | Status は `Proposed` で作成 |

### 「決めない」を記録する

私たちのADRフォーマットは、Michael Nygardによる[原典](https://cognitect.com/blog/2011/11/15/documenting-architecture-decisions)の4要素（Title / Context / Decision / Consequences）をベースにしつつ、**「決めていないこと」** セクションを追加しています。

これは [@kawasima さんの「決めない技術選定」](https://x.com/kawasima/status/2026914779865210948) がきっかけで取り入れました。この中で語られている考え方が示唆的です。

> 最終決定なんてものは存在しない。どうせ変わるものは最初から"変えられる形"にしておく。決めないんじゃなくて、決めなくていいことを決めない。

つまり、意思決定には「今決めるべきもの（後で変えにくいもの）」と「今は決めないもの（後で変わる前提のもの）」の二種類があります。そしてこの境界をはっきりさせることが、あいまいさと柔軟さを分ける線になります。

大切なのは「決めていないこと」「決めない理由」「いつ決めるか」をセットで記録しておくことです。これがないと「検討漏れ」なのか「意図的な先送り」なのかが区別できません。

```markdown
## 決めていないこと

| 項目 | 決めない理由 | いつ決めるか |
|------|------------|------------|
| キャッシュの TTL 設定 | 実トラフィックのデータが不足 | β版リリース後に計測して決定 |
| Redis のクラスタ構成 | 現時点では単一ノードで十分 | MAU 10万到達時に再検討 |
```

ここまで書いておけば、「当時は最適だった」と言って変更をやめることもできなくなります。再検討のトリガが明文化されているので、条件を満たしたら自然と見直しが始まる。ADRが過去の記録にとどまらず、未来のアクションにつながるしかけになります。

### skill 化の効果

実際、弊社のあるリポジトリでは2026年3月時点で23個のADRが蓄積されています。AIの助けで書くハードルが下がったことが大きいですが、効果は「執筆の省力化」だけではありません。

**フォーマットの統一**が最大の効果です。ADRは人によって書き方がバラバラになりがちですが、skillがテンプレートとガイドラインを強制することで、どのADRを読んでも同じ構造で情報を得られるようになりました。

また、**暗黙知の形式知化**にも効いています。「比較表を書く」「緩和策を付ける」「決めていないことを明示する」といった「良いADRを書くためのノウハウ」がskillに埋め込まれているため、ADRを書いた経験が少ないメンバーでも品質の高い記録を残せます。

## 「書いても読まれない」を解決する — 組織にADRを届けるしくみ

ADRは書くだけでは意味がありません。チームや組織に届いて初めて価値を持ちます。

### GitHub Actions による変更検知と Slack 通知

`docs/adr/` 配下のファイルがmainブランチにマージされたタイミングで、自動的にSlackへ通知を送るGitHub Actionsを設定しています。

```yaml
name: Notify ADR Changes

on:
  push:
    branches:
      - main
    paths:
      - 'docs/adr/**'

jobs:
  notify:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v6
        with:
          fetch-depth: 2

      - name: Detect changed ADR files
        id: changed
        run: |
          files=$(git diff --name-only HEAD~1 HEAD -- 'docs/adr/' || true)
          if [ -z "$files" ]; then
            echo "found=false" >> "$GITHUB_OUTPUT"
            exit 0
          fi
          echo "found=true" >> "$GITHUB_OUTPUT"

          repo_url="${{ github.server_url }}/${{ github.repository }}"
          message=""
          while IFS= read -r file; do
            name=$(basename "$file")
            url="${repo_url}/blob/main/${file}"
            message="${message}• <${url}|${name}>"$'\n'
          done <<< "$files"

          {
            echo "message<<EOF"
            echo "$message"
            echo "EOF"
          } >> "$GITHUB_OUTPUT"

      - name: Slack Notification
        if: steps.changed.outputs.found == 'true'
        uses: rtCamp/action-slack-notify@v2
        env:
          SLACK_WEBHOOK: "${{ secrets.SLACK_WEBHOOK }}"
          SLACK_TITLE: "📖 ADRが更新されました"
          SLACK_MESSAGE: |
            *変更ファイル:*
            ${{ steps.changed.outputs.message }}
          SLACK_COLOR: '#4A90D9'
          MSG_MINIMAL: true
```

ポイントは `fetch-depth: 2` と `git diff HEAD~1 HEAD` の組み合わせです。直前のコミットとの差分だけを見ることで、変更されたADRファイルだけを正確に検知しています。

### 定例のアジェンダに組み込む

通知だけでは「見たけど読まなかった」になりがちです。

弊社ではチームを横断したエンジニア全員参加のデイリー定例があります。この場のアジェンダにADRの共有枠を設けています。

しくみはシンプルです。

1. 定例に固定アジェンダとして「設計共有」の時間を設ける
2. その場でSlackのADRが通知されているチャンネルを開き、新規作成/更新されているADRを確認する
3. ADRの作成者が2〜3分で背景と決定内容を共有する
4. 参加者からの質問やフィードバックがあれば、ADRに追記する

この「通知 → アジェンダ → 共有 → フィードバック」のサイクルが回ることで、ADRはドキュメント置き場に眠る静的な文書ではなく、チームの意思決定を駆動する動的なツールになります。

さらに、この場を通じて他チームのエンジニアが自チームのADRに触れる機会が生まれます。「うちのチームでも同じ課題があった」「この選択肢は検討した？」といった横のつながりが、組織のナレッジを厚くしていきます。

## まとめ

ADRの課題とそれに対するアプローチを整理すると、以下のようになります。

| 課題 | アプローチ | 効果 |
|------|-----------|------|
| 書くのが面倒 | Claude Code skill による生成 | 執筆コストの削減・品質の統一 |
| 品質がバラつく | skill にガイドラインを埋め込む | 誰が書いても一定品質を担保 |
| 書いても読まれない | GitHub Actions + 定例での共有 | 組織全体へのナレッジ流通 |

ADRは「書く」ことがゴールではありません。書かれた意思決定が組織の中で流通し、次の意思決定に活かされてこそ価値があります。

AIの力を借りて書くハードルを下げ、自動化で届けるしくみを整え、人が集まる場で対話する。この3つがそろったとき、ADRは「書く文化」から「回る文化」へと変わります。
